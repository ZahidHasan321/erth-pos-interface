This file carries §0 governance, a one-paragraph-per-area summary of each spec area, and the companion-doc index — **read the detailed files on demand** (see "When to read which file" below).

## 0. How to use these files (governance — read first)

1. **`SPEC.md` is the spec.** Every feature, branch, and edge case lives there in plain language (§1–§6). This file is its lean index — the per-area summaries below are lossy; when a summary and `SPEC.md` differ, **`SPEC.md` wins**. Behavior not described in `SPEC.md` isn't specified — surface the gap, don't infer it from code.
2. **Spec-as-oracle.** The lifecycle test suite encodes `SPEC.md`. Test vs. code disagree → the code is the bug. Spec vs. code disagree → the spec wins (fix the code; or if the rule itself is wrong, change `SPEC.md` deliberately first, then the code). A test's expected value comes from the spec or a universal invariant (accounting identity, idempotency property) — never copied from an RPC/trigger body. A test derived from the implementation is green by construction and catches nothing.
3. **Change protocol.** A new edge case → add it to `SPEC.md`, then match code + tests. A workflow/feature change → edit `SPEC.md` first (the now-disagreeing tests reveal the blast radius), then the code. Change the spec only deliberately; never bend it to match what the code happens to do. Update this file's summary in the same change so the index stays honest.
4. **Code-decoupled.** `SPEC.md` §1–§6 describe *what* and *why* — no file paths or line numbers. Code locations live only in `IMPLEMENTATION_MAP.md` (§10), which is non-authoritative and may drift. If it disagrees with the spec, the spec wins.

**When to read which file** (each is read on demand — keyword → file):

- garment / lifecycle / order / piece_stage / brova / final / dispatch / trial / refund / cancellation / status labels / root-cause / investigation → **`SPEC.md` §2**
- cashier / payment / EOD / register / reconciliation → **`SPEC.md` §3**

- domain / entities / architecture → **`SPEC.md` §1**; shop behavior → **§5**; workshop behavior (performance, QC analytics, decisions hub, scheduling) → **§6**

## 1. Domain & architecture → `SPEC.md` §1

Dishdasha production & POS (brands ERTH/SAKKBA/QASS); two order types — **WORK** (custom tailoring) and **SALES** (pre-made shelf). **The system thinks in garments, not orders:** an order is a container, each garment row tracks one physical piece independently, and order state is derived from its garments. **Two apps, one DB** — Shop (`apps/pos-interface`) and Workshop (`apps/workshop`); the frontend never touches the DB directly — all writes go through RPCs/triggers so lifecycle rules are enforced server-side. **Brand types (§1):** brands split into **showroom** (ERTH — physical showroom, holds all stock, cashier, brova+finals, pickup-or-delivery) and **home-based** (SAKKBA/QASS — no showroom, no cashier/paid inline, no brova/finals-only, delivery-only with `home_delivery` forced, final handover on a per-brand **Delivery page**, draw fabric from ERTH's stock); one brand-set+helper drives it. **Everything is per-brand and stays in its own brand shell — the only cross-brand view is ERTH's fabric-usage-by-brand report (§4).**

## 2. The garment lifecycle (the heart) → `SPEC.md` §2

Per-garment tracking (`piece_stage`, `feedback_status`, `acceptance_status`, `location`, `trip_number`, `garment_type`, `trip_history`, `needs_investigation`). Stage chain `waiting_for_acceptance → … → completed`; terminal = `completed`/`discarded`; `soaking` parallel; `post_cutting` disabled. Key flows: brova-parking (any brova parks all finals), dispatch (`trip 0→1`), workshop receive/produce/dispatch-back, the **brova-trial branch tree** (Accept / Accept-with-Fix / Reject-Repair / Reject-Redo) with the finals-release gate, **shop-initiated redo** at the brova trial with three required-choice outcomes (replacement from our stock / from the customer's fabric / discard + promote a parked final to the new brova) — a replacement is FK-linked and created at the shop, waiting in the dispatch queue until dispatched like any garment (fabric auto-consumed + net-zero scrap waste annotation; **no `root_cause` captured at redo** — deferred to investigation; the workshop redo-priority queue is dropped), final collection (handover only — **no feedback form for finals**), alteration trips (no max), QC-fail rework (no trip increment, `alt_p`). **Feedback is the brova-trial form only — finals are not fed back; they are collected at handover (§3 cashier), not through a feedback/satisfaction form (§2.5).** A **returned Accept-with-Fix brova is likewise collect-only** (the customer already accepted; the fix comes back to hand over, not re-trial — only **Reject-Repair** re-trials on return; told apart by the persisted `acceptance_status`, and such a brova lands `ready_for_pickup` on receive, not `awaiting_trial`). Feedback is one record per brova, per trip; finals are assignment targets only (no per-final verdict/form). Feedback updates the *target spec*: a spec-correcting measurement row mints one new derived measurement, and an override section below the table lets staff assign it to parked finals and shared-measurement siblings (shared-group garments default to adopting with values pre-filled; non-shared garments default to keeping their own; the per-garment selector is fully open). Style override is finals-only, defaults to each final keeping its own, with an opt-in apply-to-all that prompts on different-collar finals. A right-side sheet shows every measurement in play with lineage. Editing is gated by production, not acceptance (Accept/Accept-with-Fix stay correctable): brova must be in the shop (it leaves on dispatch for its own fix/alteration), and the page locks read-only order-wide once any final is **in production** (workshop-started: `in_production: true`, which can occur while still `waiting_cut` — acceptance releasing finals does not itself lock). **A style change reprices the order on feedback submit:** it rewrites the garment spec **and** recomputes the style price for the garments that changed (active garment + propagated finals) using the same order-creation engine (so qallabi/designer flat styles keep their fixed price), shows the staff an old→new `order_total` delta to confirm, and rolls the delta into `order_total`/`style_charge` via the idempotent audit-only `reprice_order_styles` (never touches `orders.paid` — collection stays §3 cashier; needs_redo excluded; only the style component moves). Cancellation/refund is staff judgment with mechanical guardrails only (§2.6). Order phase derived from garments (§2.7); showroom/workshop status labels (§2.8); shared **root-cause taxonomy** (§2.9); the **repeated-returns auto-hold investigation was removed** — nothing is flagged, hidden, or blocked on repeated returns; investigation handling is being redesigned elsewhere (§2.10); **four toggle options** (`wallet_pocket`/`pen_holder`/`mobile_pocket`/`small_tabaggi` Yes/No) are a **tick mark that defaults to No** on the shop new-work-order form (un-ticked = explicit No, no confirmation block), while **QC stays blank-until-chosen and checks them both ways** (§2.11); the workshop add-garment form still requires an explicit choice. **Two categorical body measurements** live on the `measurements` row next to the numeric dimensions (required no-default; flow through measurement entry/update/feedback/QC/read-out; QC verifies both ways by equality): **`shoulder_slope`** (four fixed shapes — sloped down/up, straight, peaked) and **`collar_position`** (Up/Down/Standard, Standard = absence/null) — collar moved here from the §2.11 toggles per stakeholder direction ("collar position is part of the measurement, not an option") and is entered next to the shoulder slope (§2.12). **Order linking** (§2.13) lets a customer's confirmed WORK orders share one delivery date as a group (child `linked_order_id` → primary; group key `COALESCE(linked_order_id, order_id)`); linking is shop-only, but the workshop now **sees** it — the Production Tracker clusters a group's orders adjacently and workshop dispatch warns before stranding a linked sibling still on the workshop side (UI-only guard). **Alteration-out** (§2.14) is a separate order type (`ALTERATION`, garment `alteration`) for repairing a customer-brought **completed** garment — internal (we made it; required source-garment link) or external (another shop); intake records **only the changes** (sparse measurements + styles, **new value only — no current/baseline shown**, for both); it runs normal production; workshop **QC checks only the changed fields**; and it is **collect-only at the cashier with no feedback/trial form** (a still-wrong one is re-issued as a fresh alteration order). **Home-based brand variant (§1):** their WORK orders run the same production spine with the **brova surface removed** — garments created `final` only (no brova-parking, no trial, `acceptance_status`/`feedback_status` never written, §2.5 N/A), paid inline, `home_delivery` forced; once every garment is back at the shop `ready_for_pickup` the **whole order** is handed over on the brand's Delivery page (§5: sets `fulfillment_type: delivered`, `piece_stage: completed`, `order_phase: completed`).

## 3. Cashier, payments & EOD → `SPEC.md` §3

Cashier surface lives inside the shop shell (`/$main/cashier`, showroom brand only; tabs Pending / All Orders / Purchases — Order History and EOD are sidebar pages). The standalone `/cashier` terminal was removed; cashiering is done by shop staff/managers. Order-taker confirms `paid: 0` → order enters the cashier queue; payment is a `payment_transaction` (a trigger sums it into `orders.paid` — **never write `orders.paid` directly**); partial/installment; idempotent. **WORK cashier-processing gate:** a confirmed WORK order is pending cashier processing and **cannot dispatch to the workshop** until a cashier either **confirms-without-payment** (no money, no open register) or records a payment — either sets the one-time `work_orders.cashier_processed_at` marker (the dispatch gate; SALES/ALTERATION not gated, never pending). The cashier can **bulk-process** selected pending WORK orders: confirm-without-payment, or take payment in one **atomic all-or-nothing, idempotent** batch (per-order Advance=`work_orders.advance` / Full presets; still flows through `record_payment_transaction`, open register required; each order keeps its **own** invoice). A **lump-sum advance-first** entry seeds the per-order amounts from one amount the customer hands over (fill advances, then spread the remainder proportionally by open balance — never an equal split; UI seeding only, cashier-overridable, RPC still records explicit per-order amounts). The Pending/All-Orders lists also surface **read-only** link context: order-link groups (§2.13) badged + clustered adjacently, and Secondary customer accounts (§5) shown as "<relation> of <Primary>". Order-taker confirm persists `work_orders.advance` for ERTH (informational, not a payment). **Invoice revisions = signed-invoice content changes:** confirmation issues the invoice at revision 0 (the original signed copy); plain payments and measurement-only corrections **never** bump it; a **refund**, a **brova-trial style change** (even at unchanged price — flat qallabi/designer swaps / net-zero edits re-issue the invoice), or a **delivery-type change** (home ↔ pickup) bumps `work_orders.invoice_revision` (idempotent; a no-op change does not bump), printed as `<inv>-R<n>` for n≥1; the cashier Order-History reprints the **proper signed** `OrderInvoice`/`SalesInvoice` at the current revision while still listing every transaction. **Pickup is ungated on payment** (staff judgment). **Stock-purchase settlement (§3, Purchases tab):** a costed shop fabric/shelf restock is a non-customer expense — it mints an **unpaid payable** (`stock_purchases`, total = qty×unit_cost) the cashier settles from the Purchases tab (`stock_purchase_payments` summed into `amount_paid`/`status` by a trigger, never `orders.paid`); partial allowed, no overpay, idempotent; **cash** posts a drawer `cash_out` (open register + EOD reconciliation), **non-cash** (knet/link/bank) doesn't touch the drawer. EOD register close: the cashier opens AND closes their own session, **reopening a frozen day is allowed for any active shop user** (no longer manager-only; page-gated via RBAC, brand access still enforced server-side); reconciliation follows the cash-drawer identity `expected_cash = float + cash_payments − cash_refunds + cash_in − cash_out`; idempotent close; append-only `register_close_events`; a frozen day rejects money. Brand gate = **brand type (§1)**: the cashier is the **showroom-brand** surface (currently ERTH only) routing inline-payment vs cashier-queue; **home-based brands have no cashier** (paid inline) and complete orders on a per-brand **Delivery page** (§5) instead.

## 4. Inventory & transfers → `SPEC.md` §4

Inventory, Transfers, and Reports on each app, plus **Stocktake on the shop only**. **Fabric + shelf live only in shop stock (workshop never holds them — fabric is consumed shop-side at confirmation); accessories are the only stock that crosses sides** (carry both shop + workshop counts), so the **workshop's surfaces are accessories-only** (no fabric/shelf inventory, no fabric-usage report, no stocktake) and **transfers carry accessories only**. Each side blind to the other; crossing only via a recorded accessory transfer. Ownership: shop creates fabrics + shelf, workshop owns accessories (anything it operates — devices/supplies, not a fixed list); customer-brought fabric (`fabric_source: OUT`) is never in either stock. Append-only `stock_movements` ledger — **all changes via stamping RPCs, nothing silently unlogged**. **Cost basis (WAC):** fabric/shelf carry an `avg_cost` weighted-average cost distinct from selling price, maintained on every costed restock (first costed restock seeds it); a shop fabric/shelf restock now **requires a unit cost** and mints a cashier payable (§3). Accessories unaffected (optional cost, no payable). Order confirm rejects on insufficient stock (locks the row, never negative). Low-stock alerts (per-item threshold, **falling-edge** notification). Mandatory monthly stocktake (soft-enforced; **any active user may validate** — no longer manager-only, page-gated via RBAC) is **shop-only** — the workshop holds only a small accessory set and corrects count discrepancies with an inline Adjust. Damage/Waste (categorized reason; **no cost-threshold manager gate** — any waste-permitted user records any value, page-gated) is distinct from Adjust. Redo material waste = **net-zero annotation** (`annotated_qty` + `root_cause`) so conservation holds. Transfers are request → send → receive (**no approval gate**; bulk send atomic). RBAC table per surface. **Brand attribution (§1/§4):** a work-order fabric `consumption` row carries the **consuming brand**; since home-based brands hold no stock and draw from ERTH's shared shop pool, ERTH's **fabric report breaks consumption down by consuming brand** — the only cross-brand view; every other surface is per-brand.

## 5. Shop app (`apps/pos-interface`) → `SPEC.md` §5 · conventions `ENGINEERING.md` §11

Order-taker shell + a separate role-locked Cashier shell (§3); brand is determined by the authenticated user. **Customer accounts (§5):** Primary vs Secondary, where a Secondary is **explicitly linked to one Primary by a stored FK (`primary_customer_id`), not by a shared phone** (linked family may share or differ on mobile); account type is chosen, never silently inferred; entering a mobile already on file (matched on the **normalized national number**) **hard-blocks** and forces a choice — link as a family member of the existing Primary, or fix the number — and a Secondary always carries a linked Primary + a relation. **Home-based brands (§1):** order-taking takes payment inline, forces home delivery (no Pick-Up), and creates **finals only** (no brova); each home-brand shell carries a dedicated **Delivery page** — the home-brand analogue of the cashier handover, scoped to that brand and never shown on ERTH — that lists orders **ready to deliver** (every garment back at the shop `ready_for_pickup`) and hands over the **whole order** in one all-or-nothing action (no money handled). UI direction, typography, and tech-stack conventions live in `ENGINEERING.md` §11.

## 6. Workshop app (`apps/workshop`) → `SPEC.md` §6 · conventions `ENGINEERING.md` §12

Operational control-panel tool. **Worker team (unit) assignment is explicit, never silently defaulted.** Redo priority queue (`immediate` / `next_slot` / `parked` + `redo_parked_reason`; resume-parked re-consumes fabric). **Per-station performance scoring** (individual cutting/finishing/ironing/QC; sewing at unit level; soaking excluded); **redo performance impact attributed by `root_cause`** (no blanket factory penalty). QC analytics over the stored 1–5 ratings (`< 4` = fail), plus **per-defect team attribution**: on a QC fail the inspector tags each error row (Found→Should-be) in the fail dialog with the causing stage (cutting/sewing/finishing/ironing; soaking + QC never attributable), the responsible worker/unit is resolved from this trip's `worker_history`/`production_plan` and frozen on the attempt (`defect_attributions`, optional/manual; auto-suggestion deferred), and QC Analytics adds a **Defects by team & worker** breakdown (distinct from Performance's routing-based per-unit defect rate). **No workshop Decisions / Needs-Investigation surface — the §2.10 repeated-returns auto-hold was removed** (no garment held or blocked). Typography/color/layout/code conventions live in `ENGINEERING.md` §12.
